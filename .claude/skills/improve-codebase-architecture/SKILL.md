# 架构深化审查（Improve Codebase Architecture）

## 核心原则

**Agent 加速编码的同时也在加速软件熵。** 你不只是在修当下的问题，你还要关心代码的设计是否在正确的轨道上。

---

## 审查流程

### 阶段 1：探索 — 有机摩擦识别

遍历代码库（通过 Explore agent 或直接读取），重点关注以下五种架构异味：

#### 1. Bouncing（跳转疲劳）
理解一个概念需要访问很多小模块。

**检测信号**：
- 追踪一个 API 调用需要打开 5+ 个文件
- 相关的逻辑散落在不同目录
- 新增一个类似功能需要在 4+ 个位置添加代码

**aifriend 特定检查**：
- aifriend 有 22 个 service 模块、6 个 repository 模块——同类功能是否在对应模块内内聚？
- 路由层的 admin/ 按域拆分是否合理？有没有应该在一起却被拆开的逻辑？

#### 2. Shallowness（浅模块）
模块的公共接口几乎和实现一样复杂——它没有提供真正的抽象杠杆。

**检测信号**：
- 函数签名有 5+ 个参数，但函数体只有 10 行
- 模块的对外接口数量接近内部函数数量
- "想象删除这个模块——复杂度只是移到 N 个调用者身上"

**核心诊断工具 — 删除测试**：
想象删除这个模块。如果：
- 复杂度只是平移到调用者 → **模块太浅，删除它**
- 复杂度作为整体消失 → **模块够深，保留它**

#### 3. Misplaced Testability（测试性错位）
纯函数被提取出来仅仅是为了方便测试，真正的复杂度藏在"怎么调用它们"的地方。

**检测信号**：
- 测试覆盖了大量纯函数，但集成点、状态转换、边界条件没有被测
- 函数的测试比函数本身还长（而且是因为 mock 太多）

**aifriend 特定检查**：
- FakeSequenceConn 模拟的 SQL 是否正确反映了真实查询的行为？
- 有没有关键的业务逻辑藏在路由层的 try/except 里，没法单独测试？

#### 4. Leakage（耦合泄漏）
紧密耦合的模块跨越了它们的分层边界。

**检测信号**：
- repository 层有业务逻辑判断
- service 层直接拼接 SQL 字符串
- 路由层直接访问数据库连接（违反 `get_db_dep()` 规则）

**aifriend 特定检查**：
- 对照 CLAUDE.md 分层架构：`routers/ → services/ → repositories/ → core/ + constants/`
- 有没有反向依赖？（尤其 core/ 导入 services/ 是红线）
- repository 的方法是否只做数据访问，不掺杂业务规则？

#### 5. Untestable Code（不可测试的代码）
模块的接口让测试很难写。

**检测信号**：
- 模块在构造时就建立了外部连接（数据库、网络）
- 依赖通过 `import` 硬编码而非参数注入
- 状态分散在多个地方，测试需要设置大量前置条件

### 阶段 2：展示 — 结构化呈现

将发现的问题按以下格式呈现给用户：

```markdown
## 架构审查发现

### 发现 1：[简短标题]
**涉及文件**：
- `path/to/file1.py`（行 xx-yy）
- `path/to/file2.py`（行 xx-yy）

**问题**：[用 CONTEXT.md 中的领域术语描述问题]

**删除测试结果**：[如果适用]

**建议**：
[具体的改进方向，引用 CONTEXT.md 术语和已有的 ADR]

**收益**：
- [具体收益 1]
- [具体收益 2]
```

### 阶段 3：对话深化

用户选择一个发现后，逐个走查设计决策树：
1. 约束条件是什么？
2. 依赖关系是什么？
3. 新模块的接口长什么样？
4. 分层边界在哪里？
5. 现有测试如何保留？

当决策落地时，同步更新 CONTEXT.md（新术语）或创建 ADR（不可逆的架构决策）。

## 与 aifriend 现有规则的互补关系

本 skill 与 CLAUDE.md 中的三条质量规则互补，不重复：

| 现有规则 | 本 skill 补充 |
|----------|--------------|
| **单文件上限 1000 行** | 文件大小是症状，删除测试找到根因——是模块太浅还是不合理的拆分？ |
| **工具函数分类存放** | Bouncing 检测验证分类是否合理——用的时候是不是还是在跳转？ |
| **设计模式一致** | 检查一致性是否掩盖了设计问题——一致性不等于好设计 |

## 注意事项

- **不要为重构而重构**。只在你已经在修改该区域代码时顺便改进
- **一次只处理一个发现**。不要试图一次性重构整个代码库
- **重构前确保测试通过**。没有测试安全网不要动结构
- **高风险模块的重构必须单独开分支**（CLAUDE.md 规则）
