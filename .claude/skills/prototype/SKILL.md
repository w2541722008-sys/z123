# 原型构建（Prototype）

## 核心原则

**原型是用来扔的。** 唯一值得保留的是你从中学到的答案。构建原型之前，先明确你在回答什么问题。

---

## 两种模式

### 模式 A：终端交互式原型（逻辑/状态问题）

用于验证：业务逻辑、状态机、算法、数据转换。

**构建方式**：
- 单个 Python 脚本，放在功能附近（如 `backend/prototypes/xxx_demo.py`）
- 运行：`python3 backend/prototypes/xxx_demo.py`
- 纯内存操作，不连数据库
- 每次操作后打印完整状态
- 没有异常处理、没有测试、没有抽象

**示例场景**：
- "好感度计算逻辑在并发场景下安全吗？" → 写个脚本模拟并发调用
- "新的剧情阶段流转逻辑对吗？" → 写个脚本站遍历所有状态转换

### 模式 B：多版本 UI 原型（UI/设计问题）

用于验证：页面布局、交互流程、视觉效果。

**构建方式**：
- 在一个路由上渲染多个版本
- 通过 URL 参数切换：`?version=1`、`?version=2`
- 页面顶部浮动版本切换栏
- 所有版本共享同一个后端（如果有的话）
- 没有错误处理、没有加载状态、没有空状态

**aifriend 特定适配**：
- 前端原型放在 `frontend/prototypes/` 下
- 原生 JS + HTML，不引入框架
- 如果涉及后端 API，用最简单的路由返回 mock 数据

---

## 原型规则

1. **代码靠近最终使用位置但明确标记为临时**
   - Python 原型：文件头注释 `# PROTOTYPE — 验证完删除`
   - 前端原型：文件头注释 `<!-- PROTOTYPE — 验证完删除 -->`

2. **一条命令运行** — 不需要安装依赖、配置环境变量

3. **纯内存** — 不持久化，除非你要验证的问题本身就涉及持久化

4. **跳过工程化** — 不写测试、不处理异常、不抽象、不优化

5. **展示完整状态** — 每次操作/视图切换后，所有相关状态可见

---

## 收尾

回答完问题后：
1. **记录答案** — 写到 commit message、ADR 或注释中
2. **删除原型代码** — `rm` 掉，不要提交
3. 如果需要把原型转化为正式功能 → 新开分支，走 TDD 流程，不复用原型代码
