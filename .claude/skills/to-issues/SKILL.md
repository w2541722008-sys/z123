# 需求拆分（to-issues）

## 核心原则

**垂直切片 > 水平分层**。不要把"写所有 API"和"写所有前端"拆成两个 issue，而是每个 issue 完成一个功能的完整链路。

错误示例（水平拆分）：
- Issue 1：写所有数据库 migration
- Issue 2：写所有 API 路由
- Issue 3：写所有前端页面

正确示例（垂直切片）：
- Issue 1：角色卡好感度等级配置功能（含 migration → API → 管理后台表单 → 测试）
- Issue 2：好感度等级在聊天中的展示（含 service → 前端聊天 UI → 测试）

---

## 拆分流程

### 第一步：收集上下文

从当前对话或引用的 issue/PRD 中提取：
- 用户要解决什么问题？
- 涉及哪些领域概念？（对照 `CONTEXT.md` 术语表）
- 有没有已经存在的 ADR 约束？

### 第二步：探索代码库

快速了解当前状态：
- 哪些模块会被修改？（对照 CLAUDE.md 中的架构图）
- 是否涉及高风险模块？（admin/、billing、chat_stream 等 8 个区域）
- 现有的数据模型和 API 契约是什么？

### 第三步：草拟垂直切片

每个切片必须：
1. **横切所有层级** — schema → API → service → repository → 前端 → 管理后台 → 测试
2. **独立可验证** — 切片完成后，用户可以体验到一个完整的功能点
3. **标注类型**：
   - `AFK`（Agent 可独立完成）— 需求明确、无需人工判断
   - `HITL`（需人工介入）— 涉及 UI 设计、产品决策、安全审计

### 第四步：与用户确认

展示拆分结果，逐条确认：
- 粒度是否合适？（太大拆分、太小合并）
- 依赖关系是否正确？
- HITL/AFK 分类是否准确？

### 第五步：发布

按依赖顺序（被依赖的先发）发布到 GitHub Issues。

## Issue 模板

每个发布的 issue 使用以下模板：

```markdown
## 要构建什么
[用端到端的行为描述，避免写文件路径和代码片段，除非是关键的 schema 定义或状态机设计]

## 涉及层级
- [ ] 数据库（migration）
- [ ] API 路由
- [ ] 服务层
- [ ] 数据访问层
- [ ] 用户端前端
- [ ] 管理后台
- [ ] 测试

## aifriend 专属检查项
- **后台管理配置**：是 / 否（如果是，管理后台哪个页面需要加什么配置？）
- **数据库变更**：是 / 否（如果是，涉及哪些表/字段？）
- **影响高风险模块**：无 / [列出涉及的模块]
  - 高风险模块清单：admin/、billing.py、chat_stream/、chat_send.py、core/auth/、core/database.py、repositories/、alembic/versions/

## 验收标准
- [ ] 标准 1：可验证的行为描述
- [ ] 标准 2：覆盖正常路径
- [ ] 标准 3：覆盖边界条件
- [ ] 标准 4：测试通过

## 阻塞
- 阻塞于：#[issue 编号] 或 "无 — 可立即开始"

## 类型
`AFK` / `HITL`
```

## 拆分质量标准

好的垂直切片满足以下所有条件：
1. 用户可以直接体验到一个完整功能（哪怕很窄）
2. 不依赖其他未完成的切片（依赖关系已解决）
3. 描述足够清晰，Agent 不需要猜测需求
4. 涉及到的高风险模块已明确标注
5. 后台管理配置和数据库变更需求已明确

## 与项目规则的关系

- 遵循 CLAUDE.md **新增功能同步配置**规则：每个涉及功能变更的 issue 必须判断是否需要管理后台配置
- 遵循 **新增功能同步测试**规则：每个 issue 必须包含测试验收标准
- 遵循 **高风险模块单独开分支**规则：涉及高风险模块的 issue 标注后，实现时单独开分支
