# 文档化设计审查（Grill with Docs）

## 核心原则

**好的设计用精确的语言描述。模糊的语言暴露模糊的思考。**

本技能通过四种机制系统性地挑战你的设计方案，每次只问一个问题，等待你的反馈后再继续下一个。

---

## 审查机制

### 机制 1：术语对齐检查

对照 `CONTEXT.md` 中的领域词汇表，检查设计方案中使用的术语：

- **冲突**：你使用的词在 CONTEXT.md 中已有定义但含义不同
  - 例：你说"好感度"但描述的行为像"沉浸度" → 立即指出并要求澄清
- **歧义**：同一个词在不同上下文中含义不同
  - 例："事件"在 intimate 玩法和 scenario 玩法中触发逻辑完全不同
- **缺失**：你引入的新概念在 CONTEXT.md 中没有定义
  - → 要求给新概念下定义，确认后更新 CONTEXT.md

### 机制 2：模糊表述锐化

当你使用模糊词汇时，要求你精确化：

| 模糊表述 | 追问方向 |
|----------|----------|
| "系统自动处理" | 哪个模块？什么触发条件？什么输出？ |
| "用户可以看到" | 哪个页面？什么条件下可见？什么格式？ |
| "性能优化" | 当前耗时多少？目标耗时多少？用什么指标衡量？ |
| "差不多就行" | "差不多"的边界在哪里？什么算不合格？ |
| "和之前一样" | 和哪个"之前"一样？那个实现还在吗？ |

### 机制 3：边界条件压力测试

构造极端场景来探测试计方案的边界：

- **并发场景**：两个用户同时修改同一个角色卡的好感度，会发生什么？
- **空状态**：新用户首次进入，没有任何历史数据时，表现是什么？
- **溢出场景**：好感度达到上限/下限时，触发事件还生效吗？
- **跨玩法场景**：用户从 intimate 卡切换到 scenario 卡，数据如何隔离？
- **失败场景**：AI 模型返回错误时，用户看到什么？好感度扣不扣？

### 机制 4：代码交叉验证

当你的设计假设与代码实际情况可能冲突时，检查代码库：

- 你假设某个模块"应该"有某个能力 → 去代码里确认
- 你引用的 API 或函数签名 → 去代码里确认参数和返回值
- 你依赖的数据库字段 → 去代码里确认字段名和类型

## ADR 产出时机

并非所有决策都需要 ADR。只有满足以下**全部三个条件**的决策才创建 ADR：

1. **难以逆转** — 一旦实施，回头改的代价很大
2. **缺少上下文会感到意外** — 未来的人看到这个设计会觉得奇怪，需要知道当时为什么这么选
3. **经过了真实的权衡** — 确实有多个可选方案，不是唯一解

### ADR 模板

```markdown
# ADR-NNNN：决策标题

**日期**：YYYY-MM-DD
**状态**：提议 / 已接受 / 已废弃 / 已替代

## 背景
[什么情况下需要做这个决策？]

## 决策
[选择了什么方案？]

## 备选方案
- 方案 A：[描述 + 为什么没选]
- 方案 B：[描述 + 为什么没选]

## 后果
- **正面**：[这个决策带来的好处]
- **负面**：[这个决策带来的代价/限制]

## 相关
- 引用的 CONTEXT.md 术语
- 相关的其他 ADR
```

ADR 文件命名：`docs/adr/NNNN-英文短描述.md`（NNNN 为递增序号）。

## 审查流程

### 第一步：加载领域模型
阅读 `CONTEXT.md` 全文，理解项目的术语体系和约束。

### 第二步：逐个机制审查
按机制 1→2→3→4 的顺序，每次只提一个问题。

### 第三步：实时更新文档
- 新的术语定义 → 更新 `CONTEXT.md`
- 不可逆的重大决策 → 创建 `docs/adr/` 文件
- 术语歧义已解决 → 更新 `CONTEXT.md` 中对应条目

### 第四步：总结
审查完成后，汇总：
- 更新了哪些 CONTEXT.md 条目
- 创建了哪些 ADR
- 还有哪些开放性问题的建议

## 与项目规则的关系

- 用中文提问和讨论，技术术语保留英文
- 尊重 aifriend 的两种玩法隔离约束
- 尊重分层依赖铁律（core/ 不能导入 services/）
- 审查结果不违反 `.out-of-scope/` 中的任何边界
