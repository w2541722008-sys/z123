# 深度设计访谈（Grill Me）

## 触发限制

**本技能只在用户明确要求深度设计审查时手动触发。** 不会在日常编码、bug 修复、常规功能开发中自动激活。这是为了避免与 aifriend 的协作规则 3（"多方案时直接给出推荐，不让用户自己选"）冲突——日常工作中 Agent 应该直接决策，而不是追问用户。

## 核心原则

**把设计树的每一根树枝都走到底。** 不满足于表面的共识，追问每一个假设、每一个边界、每一个副作用。

---

## 访谈流程

### 第一步：建立设计树

从用户的设计方案中提取决策树：
- 核心目标是什么？
- 有哪些可选方案？
- 每个方案的前提假设是什么？
- 影响范围有多大？（跨哪些模块、哪些层级）

### 第二步：逐分支深入

对每个决策分支，追问：
1. **这个选择的前提是什么？** — 如果前提不成立，方案是否还成立？
2. **边界在哪里？** — 什么情况下这个方案会失效？
3. **副作用是什么？** — 对其他模块/玩法/数据有什么影响？
4. **有没有更简单的方案？** — 能不能删掉一些东西而不是增加？

### 第三步：检查与项目约束的兼容性

将设计方案对照 aifriend 的约束体系逐条检查：
- 两种玩法隔离是否被破坏？
- 分层依赖是否单向？
- 是否涉及高风险模块？
- 是否需要后台管理配置和数据库变更？
- 是否符合 `.out-of-scope/` 的边界？

### 第四步：收敛

当所有分支都被遍历、所有问题都有明确答案时，总结：
- 最终方案
- 关键决策点及理由
- 需要更新的文档（CONTEXT.md 术语、ADR）

---

## 访谈规则

1. **一次只问一个问题**。不要抛出一串问题让用户不知所措
2. **每个问题附带推荐答案**。遵守 CLAUDE.md 规则 3——给出推荐而非让用户盲选
3. **能用代码验证的就不问用户**。如果答案可以通过读代码得到，先去读代码
4. **记录关键决策**。分歧点、权衡点写下来，必要时创建 ADR
5. **知道何时停止**。当用户说"就这样"或方案已经足够清晰时，不要为了完整而继续追问

---

## 与 CLAUDE.md 协作规则的关系

- 本技能是规则 3（"直接给出推荐"）的**例外场景**——设计阶段需要充分沟通，这不矛盾
- 日常编码中仍然遵守规则 3，不因本技能的存在而改变行为
- 如果用户在一次访谈中说"不用问了直接做"，立即退出访谈模式，回到直接推荐模式
