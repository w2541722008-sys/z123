# 诊断技能（Diagnose）

## 核心原则

诊断的本质是**建立快速反馈循环**。没有反馈循环就没有诊断——你会反复猜测、反复验证、反复失败。

**最重要的规则**：不确定时停下来想清楚，不要盲目试。每试一个方向都要有明确的假设和预期结果。

---

## 第一阶段：建立反馈循环

这是整个诊断最关键的一步。你需要一个**快速的、可重复的、确定性的**信号来告诉你 bug 是否存在。

### aifriend 项目中可用的反馈方式（按优先级排序）

1. **pytest 单测**（首选）：
   ```bash
   cd backend && python3 -m pytest ../tests/unit/test_xxx.py::test_xxx -q
   ```
   优点：隔离、可重复、快速。项目中已有 FakeSequenceConn 模式隔离数据库。

2. **curl 脚本**（API 层问题）：
   ```bash
   curl -s -X POST http://localhost:8000/api/xxx \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer <token>" \
     -d '{"key": "value"}' | python3 -m json.tool
   ```
   适合：路由层问题、认证问题、请求/响应格式问题。

3. **Python REPL / pdb**（逻辑问题）：
   ```python
   import pdb; pdb.set_trace()  # 在可疑位置设置断点
   ```
   适合：服务层复杂逻辑、状态变化追踪。

4. **logging 输出**（异步/流式问题）：
   ```python
   import logging
   logger = logging.getLogger(__name__)
   logger.debug("[DEBUG-a1b2] variable_name=%s", value)
   ```
   所有调试日志必须用唯一前缀标记（如 `[DEBUG-a1b2]`），方便后续一条 grep 清理。

5. **数据库直接查询**（数据问题）：
   ```bash
   cd backend && python3 -c "
   from core.database import get_db
   with get_db() as conn:
       rows = conn.fetch_all('SELECT ...')
       print(rows)
   "
   ```

6. **前端浏览器控制台**（前端问题）：
   直接用浏览器 DevTools 的 Console 和 Network 面板观察。

7. **alembic 状态检查**（migration 问题）：
   ```bash
   cd backend && python3 -m alembic current
   cd backend && python3 -m alembic history
   ```

### 反馈循环质量标准
- **速度**：每次运行应该在几秒内出结果，超过 10 秒就值得优化
- **信号清晰**：pass/fail 必须是明确的一行输出，不是需要人工判断的模糊结果
- **确定性**：同样输入多次运行结果一致。对于非确定性 bug，目标是提高复现率而非 100% 复现

**在没有可信的反馈循环之前，不要进入下一阶段。**

---

## 第二阶段：复现

运行反馈循环，确认三件事：

1. **失败表现与用户描述一致** — 你在修的确实是用户报告的问题
2. **可以多次复现** — 至少连续 3 次出现同样的失败
3. **捕获了确切的症状** — 你有能力在修复后验证问题消失

如果无法复现：检查环境差异（Python 版本、依赖版本、数据库状态、环境变量）。

---

## 第三阶段：建立假设

在动手改代码之前，列出 **3-5 个有优先级的、可证伪的假设**。

每个假设必须包含一个**预测**：
- "如果 X 是原因，那么改 Y 会让 bug 消失"
- "如果 X 是原因，那么改 Z 会让 bug 更严重"

格式示例：
```
假设 1（最可能）：好感度计算在并发请求下使用了过期数据
  → 预测：如果加锁保护好感度读写，错误会消失
  → 预测：如果人为创建并发场景（两个请求同时修改好感度），错误率会翻倍

假设 2：prompt 组装时 token 截断位置不正确
  → 预测：如果禁用 token 截断，输出会包含完整内容
```

**在做任何修改之前，先把假设列表展示给用户。** 用户可能有领域知识帮你重新排序。

---

## 第四阶段：插桩验证

每次只测试一个假设。每次只改一个变量。

### 验证方法（按推荐度排序）
1. **pdb 断点** — 直接看中间状态，不需要改代码
2. **临时 assert** — `assert value == expected, f"Got {value}"`
3. **tagged logging** — `logger.debug("[DEBUG-a1b2] key=%s", val)`，每条日志有唯一前缀
4. **性能问题** — 用 `time.perf_counter()` 或 Python profiler 测量，不要用日志估计

### 关键规则
- **每次只改一个变量**。改两个就是同时跑两个实验，结论无法归因
- 如果没有得到预期结果，**不要继续改**。回头重新审视假设是否正确

---

## 第五阶段：修复 + 回归测试

### 先写回归测试
- 测试必须覆盖**真实的 bug 模式**，不是模拟的简化版本
- 测试应该在 bug 出现的那一层编写（路由层的问题就在路由测试中覆盖）
- 如果找不到合适的测试接入点，**这是一个架构发现** — 说明代码的可测试性有问题

### 再修复
- 写最小化的修复代码，不要顺手重构
- 观察回归测试通过

### 最后回验证
- 回到第一阶段的反馈循环，在原始未缩小的场景中再跑一次
- 确认原始 bug 确实消失了

---

## 第六阶段：清理 + 复盘

### 清理
1. 删除所有带 `[DEBUG-xxxx]` 前缀的插桩代码：`grep -rn "\[DEBUG-" backend/`
2. 删除临时脚本和测试文件
3. 确认原始复现场景不再出错

### 复盘（写入 commit message）
在 commit message 中回答三个问题：
1. 根因是什么？（一句话）
2. 为什么之前的测试没有抓到？（如果是测试盲区，需要补充测试覆盖）
3. 类似的 bug 还可能在哪里出现？

**最后问自己**：什么措施能防止这类 bug 再次出现？如果答案是架构层面的改进，建议单独开 issue 跟踪，**不要在这次修复中顺手重构**。

---

## 与项目规则的关系

执行诊断时须遵守 aifriend 项目规则：
- 高风险模块（admin/、billing、chat_stream 等）修改时单独开分支
- 修改 SQL 后同步更新对应测试的 FakeQueryResult
- 错误信息用中文，技术细节保留英文
- 诊断过程本身不违反 `.out-of-scope/` 中的任何边界约束
