# ADR-0001：两种玩法隔离架构

**日期**：2025-05-01（推定，基于代码历史）
**状态**：已接受

## 背景

aifriend 支持两种截然不同的 AI 角色交互模式：
- **对话陪伴（intimate）**：侧重情感连接，追踪好感度，使用人生档案，重视长期记忆
- **剧情沙盒（scenario）**：侧重剧情推进，追踪沉浸度，不使用人生档案，使用剧情类型

早期的代码尝试用一套统一的逻辑处理两种模式，但很快发现两种玩法在数据模型、prompt 构建、状态追踪、事件触发等方面的差异大到无法用简单的 if/else 覆盖。

## 决策

在系统中建立"两种玩法隔离"作为最高级别的架构约束：

1. **数据隔离**：intimate 使用 `life_profile_json`（人生档案），scenario 使用 `scenario_type`（剧情类型）。两种卡类型不共享对方专属字段
2. **逻辑分叉**：`prompt_assembler.py` 根据 `card_type` 选择完全不同的 prompt 构建路径
3. **追踪系统独立**：intimate 的好感度系统和 scenario 的沉浸度系统各自独立运作，不交叉
4. **UI 动态适配**：管理后台根据 `card_type` 动态显示/隐藏字段，避免配置混乱
5. **枚举稳定性**：`card_type` 的枚举值（intimate/scenario）作为不可变常量，受 `.out-of-scope/` 保护

## 备选方案

- **方案 A：统一模式** — 用一个通用的"关系值"替代好感度和沉浸度
  - 未选原因：两种模式的交互设计目标完全不同——一个是建立情感纽带，一个是推进故事剧情。强行统一会让两个方向都做不好

- **方案 B：插件式架构** — 每种玩法作为可插拔的模块
  - 未选原因：初期只有两种玩法，插件架构的抽象成本高于收益。如果未来玩法类型超过 3 种，此方案值得重新评估

## 后果

### 正面
- 两种玩法可以各自独立演进，互不干扰
- 新增 intimate 功能不会意外影响 scenario，反之亦然
- 代码结构清晰，新人能快速理解"这里有两个世界"

### 负面
- 某些通用功能（如聊天流式响应）需要在两个路径中重复适配
- 新增第三种玩法时需要修改多个分叉点
- 管理后台需要维护两套字段显示逻辑
