### 误判纠正

原先报告 `chat_stream_service.py` 中有 15 个死代码函数。核实后发现判断颠倒了：
- `chat_stream_service.py`（531行）是**活跃文件**，被路由层直接引用
- `chat_stream_infra.py`（180行）和 `chat_stream_persist.py`（181行）才是**提取后从未接入**的孤立文件
- 这是一次未完成的重构

## 阶段一：上线前必须修复（预计 1.5 天）

### 1.1 admin 角色端点加认证鉴权 [15 分钟]

**文件**：`backend/routers/admin/characters_core.py`

**当前**（第 17 行）：
```python
router = APIRouter(tags=["admin"])
```

**修改为**（与其他 6 个 admin 子路由一致）：
```python
router = APIRouter(dependencies=[Depends(get_admin_user)], tags=["admin"])
```

**影响**：堵住 `GET /api/admin/characters` 和 `GET /api/admin/character/{id}` 两个未认证端点的数据泄露。

### 1.2 前端 XSS 修复 [10 分钟]

**文件 1**：`frontend/modules/app.js` 第 78-82 行
- `${err.message}` → `${escapeHtml(err.message)}`
- 该文件已 import escapeHtml

**文件 2**：`frontend/admin/js/char-list.js` 第 83 行
- 同样的修复

### 1.3 聊天历史加分页 [30 分钟]

**后端**：
- `repositories/chat_repository.py`：`get_chat_history()` 增加 `limit`/`offset` 参数，默认 50 条
- `routers/characters.py`：`/chat/history` 端点接收 `page`/`page_size` 查询参数，返回 `has_more` 字段

**前端**：
- `frontend/modules/chat.js`：历史消息改为分页加载，滚动到顶部自动加载更早消息

### 1.4 请求体大小限制 [20 分钟]

**文件**：`backend/main.py`
- 在安全响应头中间件之后添加 ASGI body 大小检查中间件
- 限制 5MB，超额返回 413

### 1.5 修复失败测试 [1 小时]

**文件**：`tests/services/test_character_session_service.py`
- 根因：service 新增 `_resolve_chat_clear_greeting()` 查询 characters 表，FakeSequenceConn 未配置
- 修复：补充 fake 返回数据

## 阶段二：上线后一周内（预计 3 天）

### 2.1 清理 chat_stream 重构残留 [15 分钟]

**实际方案**：直接删除两个死文件
- `chat_stream_infra.py`（180行）和 `chat_stream_persist.py`（181行）的函数与 `chat_stream_service.py` 完全重复
- 两个文件零代码引用，`chat_stream_service.py` 仅 531 行无需拆分
- 删除后 843 测试全部通过

### 2.2 admin/billing 频率限制 [2 小时]

- `routers/admin/_router.py`：路由器级注入限流依赖
- `routers/billing.py`：`POST /billing/orders` 加用户级限流（30 秒 3 次）
- 复用现有 `services/rate_limit.py`

### 2.3 chat.js 拆分 [4 小时]

1258 行 → 拆为 3 个文件 + 1 个入口：
- `chat-stream.js`（~400 行）：SSE 连接、流式状态机、断线重连
- `chat-render.js`（~450 行）：气泡渲染、Markdown、时间线
- `chat-actions.js`（~400 行）：Regenerate、Continue、重试
- `chat.js`（~100 行）：模块协调入口

### 2.4 请求追踪 ID [1 小时]

- `main.py` 中间件生成 `X-Request-ID`（uuid4 前 8 位）
- 注入到 logging extra 字段

## 阶段三：上线后一月内（预计 2 周）

### 3.1 长函数拆分（prompt_assembler + character_state）[7 小时]

- `build_layered_chat_messages`（388 行）拆为 5-6 个阶段函数
- `apply_state_delta`（250 行）拆为独立处理器

### 3.2 桌面端响应式适配 [3 小时]

- 增加 768px/1024px 断点
- style.css 分拆为 base/layout/chat/components

### 3.3 JWT 安全增强 [4 小时]

- access token（15min）+ refresh token（30d）双 token 模型
- refresh token 绑定设备指纹
- 支持踢出其他设备

### 3.4 AI 模型熔断器 [3 小时]

- Circuit Breaker 状态机（CLOSED→OPEN→HALF_OPEN）
- OPEN 状态直接返回 503，不等待超时

### 3.5 消息全文搜索 [4 小时]

- PostgreSQL tsvector + GIN 索引
- 新增搜索 API 端点

### 3.6 .env.example 模板 [2 小时]

- 列出所有必需配置项（含类型和示例值）
- APP_SECRET 支持环境变量优先于文件
