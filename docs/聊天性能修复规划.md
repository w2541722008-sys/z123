## 最终确认的 4 个问题

### 问题 1：前端 DOM 节点无上限增长 🔴 严重

**根因**：`loadEarlierMessages()` 每次点击追加 50 条消息的 DOM 节点到 `#chat-messages` 容器，无任何回收或上限机制。用户连续点击 10 次 "加载更早的消息" → 500+ 消息 → 2500+ DOM 节点。移动端 2000 节点是流畅滚动的临界点。

**调用链**：
```
loadEarlierMessages() [chat.js:243]
  → API.getHistory(page) → 返回 50 条
  → 每条调用 appendMsg() → createMessageRow() + createMsgAvatar() + createMessageBubble() + createMessageActionButtons()
  → 所有 DOM appendChild 到容器
  → 永不回收
```

**修复方案**：DOM 节点上限 + 淘汰机制

在 `appendMsg()` 中增加 DOM 计数，超过阈值（200 条消息）时移除最早 50 条的 DOM 节点，替换为单个占位 div：

```
chat.js 新增：
  const MAX_DOM_MESSAGES = 200;
  const PRUNE_COUNT = 50;

chat-render.js appendMsg() 新增：
  创建消息 DOM 后，检查 #chat-messages 子节点数
  超过 MAX_DOM_MESSAGES 时：
    删除前 PRUNE_COUNT 个 .msg-row 节点
    在最顶部插入占位 div（显示 "↑ 更早的消息已折叠，点击展开"）
    点击占位 div → 重新从 API 加载被淘汰的消息
```

**回归风险**：低。`loadEarlierMessages` 的滚动位置恢复逻辑（基于 `getBoundingClientRect`）依赖 `btn` 或 `box.firstChild` 作为锚点，占位 div 可作为新的锚点。

**文件变更**：`chat.js`（+20 行）、`chat-render.js`（+15 行）

### 问题 2：`ChatState.history` 数组无上限增长 🔴 严重

**根因**：`loadEarlierMessages()` 使用 `ChatState.history = [...olderMessages, ...ChatState.history]` 不断在前面追加，数组随翻页无限增长。

**history 数组的所有消费者**：
| 位置 | 操作 | 是否需要全量 |
|------|------|-------------|
| `chat.js:345` | `slice(-8)` 给游客模式 | 仅需最近 8 条 |
| `chat-stream.js:142-148` | 从尾部向前搜索 message_id | 仅需最近 ~10 条 |
| `chat-stream.js:22-23` | push 新消息 | 尾部追加 |
| `chat.js:374` | getter 暴露给外部 | 无外部依赖 |

**结论**：没有任何功能需要全量 history 数组。300 条上限完全安全。

**修复方案**：`ChatState.history` 最多保留 300 条。超出时从头部裁剪：

```javascript
// chat.js loadEarlierMessages() 中，拼接后裁剪
ChatState.history = [...olderMessages, ...ChatState.history];
if (ChatState.history.length > 300) {
    ChatState.history = ChatState.history.slice(-300);
}

// chat-stream.js appendLocalConversation() 中
ChatState.history.push(...);
if (ChatState.history.length > 300) {
    ChatState.history = ChatState.history.slice(-300);
}
```

**回归风险**：极低。所有 consumer 只访问尾部数据。

**文件变更**：`chat.js`（+3 行）、`chat-stream.js`（+3 行）

### 问题 3：前端缺少全文搜索入口 🟡 中等

**根因**：后端 `GET /api/chat/search` 已完整实现（tsvector + GIN 索引 + 中文分词），但前端无任何搜索 UI。

**修复方案**：在聊天顶栏 "···" 菜单右侧添加搜索图标，点击弹出搜索面板：

1. **搜索图标**：`chat-topbar-right` 同级增加 `<span class="chat-search-btn" onclick="Chat.openSearch()">🔍</span>`
2. **搜索面板**：模态覆盖层，包含搜索输入框 + 结果列表 + 分页
3. **结果交互**：点击搜索结果 → 关闭搜索面板 → 加载该消息附近的上下文 → 滚动到目标消息

新增独立模块 `frontend/modules/chat-search.js`（~200 行），在 `chat.js` 中接入。

**API 对接**：
```
GET /api/chat/search?q=关键词&character_id=xxx&page=1&page_size=20
  → { query, results: [{id, role, content, character_id, created_at, rank}], total, page, page_size, has_more }
```

**回归风险**：低。纯增量功能，不改动现有代码路径。

**文件变更**：`chat-search.js`（新建 ~200 行）、`chat.js`（+10 行）、`index.html`（+3 行）、`style/components.css`（+60 行）

### 问题 4：chat_messages 重复索引 🟢 低

**根因**：001 号迁移创建了 `idx_chat_messages_user_char (user_id, character_id)`，011 号迁移又创建了 `idx_chat_messages_search_user_char (user_id, character_id)`，两个索引完全重复。

**影响**：每次 INSERT chat_messages 需维护两个相同的 B-tree 索引，写入性能略降，存储空间浪费。

**修复方案**：新增 012 号迁移，删除重复索引：

```sql
DROP INDEX IF EXISTS idx_chat_messages_search_user_char;
```

保留 `idx_chat_messages_user_char`（001 号创建），因为它的命名更通用，且与 `idx_chat_messages_summarized`、`idx_chat_messages_created_at` 命名风格一致。

**回归风险**：零。DROP 的是完全冗余的索引，查询优化器总是可以使用另一个同名索引。

**文件变更**：`backend/alembic/versions/012_drop_duplicate_index.py`（新建 ~35 行）
